# Story design

Story design decides what the video says, the beat order, and the job of each beat. It does not decide CSS, exact animation curves, or renderer code.

## Extract product truth

Name the audience, pain/desire, promise, product role, proof, and CTA from `BRIEF.md` and `capture/inventory.json`. Do not invent proof absent from the site, recipe, or registered assets.

Build around one promise, not the website navigation order. A website is organized for browsing; a short video is organized for persuasion.

## Choose one arc

- `pas`: hook -> pain -> agitation -> solution -> product -> proof -> CTA.
- `future_pacing`: desired future -> product -> mechanism -> outcome -> CTA.
- `demo_loop`: question -> product -> input -> action -> result -> benefit -> CTA.
- `bab`: before -> after -> bridge -> steps -> result -> CTA.
- `feature_benefit_cascade`: category hook -> feature -> benefit repeated -> climax -> CTA.

## Beat rules

One beat has one role. The first 3–5 seconds create tension, curiosity, or desire and state the viewer outcome. Avoid opening with a generic company description.

For product UI, use consecutive beats:

```text
input -> user action -> product response -> result -> viewer benefit
```

Do not compress that into a single browser scene if the narration contains multiple meaningful cues.

## Voiceover shape

Write short cues that give the visual plan timing anchors. Prefer:

```text
上传现场图。选择行业。点击生成。结果直接落在真实空间里。
```

over one long clause containing the same facts. Clear phrase boundaries let visual windows reveal exactly when each phrase is spoken.

Blueprint ids are candidate shot shapes, not reasons to invent or reorder story beats. Story truth wins.
